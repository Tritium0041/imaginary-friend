# 《时空拍卖行》桌游 Agent 系统开发计划 (基于 Claude Agent SDK & GM 主控架构)

## 1. 技术栈选型

为了充分发挥大语言模型在规则理解和流程编排上的优势，采用以下技术栈：

| 模块 | 技术选型 | 说明 |
| --- | --- | --- |
| **后端语言** | Python 3.11+ | 适合快速开发、逻辑处理以及与 Claude Agent SDK 的无缝集成 |
| **Agent 框架** | Claude Agent SDK (Python) | 提供内置的 Agent 循环、多轮会话管理 (`ClaudeSDKClient`)、子 Agent 编排 (`AgentDefinition`) 以及原生的 MCP 工具支持 |
| **Web 框架** | FastAPI | 提供轻量级、高性能的异步 API 接口，支持 WebSocket（用于实时对话和状态推送） |
| **前端界面** | React + Tailwind CSS | 构建简洁的 Web UI，或者初期使用 Streamlit/Gradio 快速搭建原型 |
| **状态管理** | 内存字典或 SQLite | 作为"哑"数据存储，仅负责记录数值和物品归属，不包含复杂的业务逻辑 |

## 2. 系统架构设计：GM 主控模式

系统整体分为三层：表现层、Agent 逻辑层和状态存储层。

**核心理念：代码只做存储，GM 决定一切。**
传统的桌游电子化通常由代码状态机控制流程，而本系统将**流程推进、规则裁定、阶段切换的决策权全部交给 GM Agent**。代码层退化为一个"数据库"和"基础工具箱"，GM Agent 才是真正理解《时空拍卖行》规则的大脑。

## 3. 模块拆解与核心功能

### 3.1 状态存储与工具层 (State Store & Tools)

这一层不包含游戏流程逻辑，只提供数据读写和基础的数值计算，通过 `@tool` 装饰器封装为 MCP 工具供 GM Agent 调用。

- **数据模型**：定义玩家实体（资金、VP、手牌、文物）、全局实体（回合数、稳定性、时代倍率、公共拍卖区）。
- **原子操作工具**：
  - `get_game_state`: 获取当前所有公开和私有（对 GM 而言）的状态。
  - `update_player_asset`: 修改玩家的资金或 VP。
  - `transfer_item`: 在玩家、公共区、系统仓库之间转移文物或卡牌。
  - `update_global_status`: 修改稳定性、时代倍率或当前游戏阶段。
  - `record_sealed_bid`: 记录玩家的暗拍出价（供 GM 统一揭示）。

### 3.2 GM Agent 模块 (游戏大脑)

GM Agent 是整个系统的核心，通过 `ClaudeSDKClient` 维持整局游戏的持久化会话。系统启动时，会将《时空拍卖行》的完整 Markdown 规则书作为 System Prompt 注入给 GM。

- **流程驱动**：GM 根据规则书和当前状态，自主决定现在是"挖掘阶段"还是"拍卖阶段"，并主动呼叫相应的 Player Agent 或真实玩家行动。
- **规则裁定与结算**：
  - 玩家出价后，GM 判断其资金是否足够。
  - 拍卖结束后，GM 根据规则计算谁是赢家，并调用 `transfer_item` 和 `update_player_asset` 进行结算。
  - 发生时空震荡时，GM 计算惩罚并执行。
- **异常处理**：当玩家（真实玩家或 Agent）提出非法操作时，GM 直接在对话中驳回并要求重新行动。遇到规则书未明确的边缘情况，GM 暂停游戏并向真实玩家提问裁定。

### 3.3 Player Agent 模块 (子 Agent)

Player Agent 模拟真实玩家的对手，利用 Claude Agent SDK 的 `AgentDefinition` 作为子 Agent 挂载。

- **身份与策略**：从预设的身份库中随机抽取名称、说话风格和策略倾向，注入到子 Agent 的 `prompt` 中。
- **独立思考**：作为子 Agent，它们在自己的独立上下文中思考和决策，不会污染主 GM 的上下文。
- **交互方式**：Player Agent 不直接修改游戏状态，而是通过自然语言（或特定的结构化输出）向 GM 表达自己的行动意图（例如："我出价 15 购买那件古代文物"），由 GM 审核并执行。

## 4. 开发阶段规划

开发过程分为四个主要阶段，逐步实现系统的完整功能。

### 阶段一：状态存储与基础工具开发 (预计 1 周)

本阶段的目标是搭建"哑"数据层。定义游戏所有实体的数据结构（字典或 Pydantic 模型）。实现基础的 CRUD 工具（如转移物品、修改数值），并使用 Claude Agent SDK 的 `@tool` 装饰器将其封装为进程内 MCP 服务器。编写简单的测试脚本，确保工具调用能正确修改数据。

### 阶段二：GM Agent 核心循环与规则注入 (预计 2 周)

本阶段是整个项目的核心。将《时空拍卖行》规则书转化为适合 LLM 阅读的 Prompt。配置 `ClaudeSDKClient` 启动 GM Agent。通过纯文本交互，测试 GM 是否能正确引导游戏流程（如：宣布游戏开始 -> 抽取文物 -> 询问玩家出价 -> 结算拍卖 -> 进入下一阶段）。此时可以由开发者手动扮演所有玩家，测试 GM 的裁判和流程控制能力。

### 阶段三：Player Agent 接入与多 Agent 联调 (预计 1 周)

使用 `AgentDefinition` 定义 Player Agent。将 Player Agent 接入 GM 的主循环中。实现 GM 呼叫 Player Agent、Player Agent 思考并回复、GM 解析回复并执行状态更新的完整闭环。调整 Player Agent 的 Prompt，使其行为符合设定的策略倾向。

### 阶段四：Web UI 开发与体验优化 (预计 1-2 周)

开发基于 FastAPI 的后端接口，支持 WebSocket 实时通信。开发前端界面，包含游戏桌面（展示公共信息、拍卖区）、玩家面板（展示手牌、资产）和聊天窗口（展示 GM 的播报和 Agent 的对话）。进行全链路联调，优化 Prompt 以减少 GM 的幻觉，确保游戏流程顺畅。
